# Анализ проекта `vk-mini-app_full`

## Назначение документа
Этот документ собирает в одном месте обзор проекта, его архитектурную карту, каталог backend API, доменную модель и список архитектурных проблем в порядке приоритета.

Документ основан на изучении директорий:

- `.agent`
- `backend`
- `docs`
- `vk_radar_app`

## Что это за проект
Репозиторий представляет собой VK Mini Apps-проект с уже работающей связкой:

1. `vk_radar_app` выступает как клиентское мини-приложение на `React + Vite + VKUI + VK Bridge`.
2. `backend` выступает как API-слой на `Express + TypeScript + Prisma + PostgreSQL`.
3. Backend хранит пользователей, посты, сообщества и радар-пины, а также интегрируется с VK API.
4. `docs` содержит не внутреннюю документацию проекта, а локальную базу знаний по платформе VK.
5. `.agent` хранит инструкции и навыки для ИИ-агента, а не код runtime-части продукта.

По текущему состоянию проект выглядит как рабочий прототип или ранняя продуктовая версия community-based VK Mini App вокруг двух главных сценариев:

- карта событий и объявлений;
- модерация и публикация постов от имени сообщества.

## Карта репозитория

### `.agent`
Каталог с правилами и навыками для агента.

Что важно:

- `AGENT_RULES.md` задает общие правила поведения агента;
- `workflows/zapomni.md` и `workflows/zapishi.md` описывают специальные команды;
- `skills/*` содержит reusable-навыки по brainstorming, debugging, frontend, fullstack, UI/UX, документам и мультимедиа.

Практический вывод:

- `.agent` не участвует в runtime проекта;
- он влияет на процесс работы агента;
- часть файлов выглядит как общая библиотека навыков, не полностью привязанная к текущему репозиторию.

### `backend`
Основной серверный слой проекта.

Ключевые файлы:

- `backend/src/index.ts` — bootstrap Express-приложения;
- `backend/src/apps/vk_post_app/routes.ts` — почти вся backend-логика проекта;
- `backend/src/shared/db.ts` — Prisma-клиент;
- `backend/src/shared/middleware/vkAuth.ts` — валидация launch params и подписи VK Mini Apps;
- `backend/src/utils/vkApi.ts` — вызовы VK API, публикация постов и поиск городов;
- `backend/prisma/schema.prisma` — модель данных.

### `vk_radar_app`
Клиентское VK Mini App.

Ключевые файлы:

- `vk_radar_app/src/main.tsx` — инициализация приложения и `VKWebAppInit`;
- `vk_radar_app/src/AppConfig.tsx` — интеграция VK Bridge, темы и роутера;
- `vk_radar_app/src/App.tsx` — shell приложения;
- `vk_radar_app/src/routes.ts` — panel routing;
- `vk_radar_app/src/utils/api.ts` — общий HTTP-клиент к backend;
- `vk_radar_app/src/panels/*` — основные экраны.

### `docs`
Сейчас это в основном локальная база знаний по платформе VK:

- `docs/vk-knowledge-base/offline-docs` — офлайн-зеркало документации `dev.vk.com`;
- `docs/vk-knowledge-base/spider` — локальный scraper;
- `docs/vk-knowledge-base/mcp-server` — локальный MCP-сервер для подтягивания VK docs.

Практический вывод:

- `docs` почти не содержит hand-written документацию о самом проекте;
- основной смысл этой директории — локальный справочник по внешней платформе VK.

## Архитектурная карта

### Общий поток данных

1. Пользователь открывает VK Mini App внутри VK.
2. `vk_radar_app` получает launch params из URL.
3. Клиент отправляет эти launch params на backend в `Authorization: Bearer ...`.
4. `backend` валидирует подпись, извлекает:
   - `vk_user_id`
   - `vk_group_id`
   - `vk_viewer_role`
5. Дальше backend:
   - читает и пишет данные в PostgreSQL через Prisma;
   - создает/редактирует посты и радар-пины;
   - управляет настройками сообщества;
   - вызывает VK API для публикации, уведомлений и справочных данных.

### Основная архитектурная связка

- фронтенд: `vk_radar_app`
- backend API: `backend`
- база данных: PostgreSQL через Prisma
- внешняя платформа: VK API + VK Mini Apps + VK Bridge

### Архитектурный профиль
Проект устроен как компактный монолит:

- один frontend-app;
- один backend-сервис;
- одна ключевая роут-группа `vk_post_app`;
- одна централизованная Prisma-схема;
- без отдельного слоя сервисов/репозиториев/очередей.

Это делает проект понятным для быстрого старта, но уже создает проблемы масштабируемости и поддержки.

## Frontend: устройство `vk_radar_app`

### Навигация
Роутинг построен на `@vkontakte/vk-mini-apps-router` и `createHashRouter`.

Основные панели:

- `radar`
- `list`
- `my_ads`
- `create_ad`
- `profile`
- `moderation`
- `settings`

Главный navigation shell собирается в `vk_radar_app/src/App.tsx` через `Epic`, `View` и `Tabbar`.

### Основные экраны

#### `Radar`
Главный экран приложения.

Что делает:

- инициализирует VK Maps SDK;
- загружает пины с backend;
- показывает маркеры на карте;
- определяет стартовую геопозицию;
- позволяет создавать новые пины;
- позволяет администратору привязать центр карты к сообществу;
- использует `localStorage` для личных настроек позиции.

Это самый насыщенный и самый перегруженный экран приложения.

#### `List`
Лентовый экран.

Сейчас фактически показывает список радар-пинов, хотя интерфейс намекает на более широкую смешанную ленту.

#### `MyAds`
Экран пользовательских сущностей.

Что делает:

- загружает собственные пины;
- загружает собственные посты;
- объединяет их в общий список;
- позволяет удалять и редактировать записи.

#### `CreateAd`
Форма создания объявления.

Сценарий:

- пользователь вводит данные;
- клиент отправляет `POST /posts`;
- пост создается как материал на модерацию.

#### `Profile`
Профиль пользователя и точка входа в админский инструментарий.

Там сосредоточены:

- базовая информация о пользователе;
- переход в модерацию;
- переход в настройки сообщества.

#### `Moderation`
Экран модерации постов.

Что делает:

- получает очередь постов на утверждении;
- позволяет approve/reject;
- для уже одобренных постов подтягивает статистику из VK.

#### `AppSettings`
Экран настройки конкретного сообщества.

Что делает:

- загружает текущие настройки и список админов;
- позволяет выбрать город и дежурного администратора;
- управляет расписанием публикаций;
- сохраняет конфигурацию сообщества.

### Data flow на фронтенде
На клиенте почти нет централизованного state management.

Типовой паттерн:

1. panel mounts;
2. `useEffect` вызывает `vkFetch(...)`;
3. данные кладутся в локальный `useState`;
4. изменения отражаются через локальное обновление состояния или повторный fetch.

Плюсы:

- низкий порог входа;
- минимальный объем инфраструктурного кода.

Минусы:

- дублирование data fetching;
- нет кэша и нормальной invalidation-модели;
- ошибки и loading размазаны по экранам;
- сложные панели, особенно `Radar`, быстро разрастаются.

## Backend: устройство `backend`

### Runtime entrypoint
`backend/src/index.ts`:

- загружает `.env`;
- создает Express app;
- включает CORS и JSON parsing;
- раздает `backend/uploads` как статические файлы;
- монтирует router на `/api/vk_post_app`.

### Главный backend-модуль
`backend/src/apps/vk_post_app/routes.ts` фактически является центром всей серверной логики.

В нем сосредоточены:

- community endpoints;
- post endpoints;
- radar endpoints;
- upload handling;
- schedule slotting;
- вызовы VK API;
- часть логики авторизации;
- работа с Prisma.

Это главный файл, который сильнее всего определяет реальную архитектуру backend.

### Auth-модель
`backend/src/shared/middleware/vkAuth.ts` работает так:

1. читает `Authorization: Bearer ...`;
2. интерпретирует bearer как строку launch params;
3. валидирует `sign`;
4. записывает в запрос:
   - `vk_user_id`
   - `vk_group_id`
   - `vk_viewer_role`

Затем все защищенные endpoint’ы используют этот контекст для определения пользователя, сообщества и роли.

### VK API integration
`backend/src/utils/vkApi.ts` инкапсулирует общение с VK API.

Что уже реализовано:

- универсальный `callVkApi(...)`;
- поиск городов через `database.getCities`;
- публикация постов в сообщество через:
  - `photos.getWallUploadServer`
  - `photos.saveWallPhoto`
  - `wall.post`

### Upload-модель
Файлы складываются локально в `backend/uploads`.

Backend затем:

- раздает их как статику;
- берет локальные пути при публикации фото в VK.

Это простой и рабочий подход для разработки, но он пока не выглядит production-ready.

## Каталог backend API

Все endpoint’ы ниже относятся к `/api/vk_post_app`.

### Публичные endpoint’ы

#### `GET /community/:id`
Назначение:

- получить настройки сообщества;
- используется фронтендом для привязки радара к конфигу группы.

#### `GET /ping`
Назначение:

- техническая проверка доступности сервиса.

### Radar endpoint’ы

#### `GET /radar/pins`
Назначение:

- получить активные пины на карте.

Поведение:

- фильтрует по `expiresAt > now`;
- возвращает `author` и `post`.

#### `POST /radar/pins`
Назначение:

- создать ручной пин на карте.

Поведение:

- извлекает текущего пользователя;
- создает `User`, если он еще не создан;
- создает `RadarPin` со сроком жизни 48 часов.

#### `GET /radar/pins/my`
Назначение:

- получить пины текущего пользователя.

#### `PATCH /radar/pins/:id`
Назначение:

- изменить заголовок и описание пина.

Права:

- автор или администратор.

#### `DELETE /radar/pins/:id`
Назначение:

- удалить пин.

Права:

- автор или администратор.

### Post endpoint’ы

#### `GET /posts`
Назначение:

- лента одобренных постов для текущего сообщества.

#### `GET /posts/my`
Назначение:

- посты текущего пользователя в текущем сообществе.

#### `GET /posts/moderation`
Назначение:

- очередь модерации по статусу.

Текущее ограничение:

- в коде прямо указано, что строгая серверная проверка роли администратора здесь еще не доведена до конца.

#### `POST /posts`
Назначение:

- создать новый пост.

Поведение:

- создает или находит `User`;
- создает пост для одной или нескольких групп;
- гарантирует существование `Community`;
- может автоматически создать `RadarPin`;
- может отправить уведомления администраторам.

Это одна из самых важных ручек в системе.

#### `PATCH /posts/:id`
Назначение:

- изменить текст и тег поста.

Особенность:

- после редактирования статус снова сбрасывается в `pending`.

#### `DELETE /posts/:id`
Назначение:

- удалить пост.

Права:

- автор или администратор.

#### `PATCH /posts/:id/status`
Назначение:

- управлять модерацией.

Поведение:

- принимает `pending | approved | rejected`;
- при approve может автоматически подобрать слот публикации;
- может инициировать реальную публикацию поста в VK;
- сохраняет `vkPostId`.

Это главная orchestration-ручка проекта.

#### `PATCH /posts/swap-time`
Назначение:

- поменять местами время публикации двух постов.

Особенность:

- сейчас меняет не только `publishDate`, но и `createdAt`.

### Community endpoint’ы

#### `POST /community/token`
Назначение:

- сохранить токен сообщества.

#### `PATCH /community/:id`
Назначение:

- обновить настройки сообщества.

Что меняется:

- город и регион;
- координаты;
- дежурный администратор;
- список уведомляемых администраторов;
- accepted tags;
- параметры расписания;
- флаг cross-posting.

Дополнительно:

- backend пытается подтянуть название и аватар сообщества через VK API.

#### `GET /community/city/:cityId`
Назначение:

- получить список партнерских сообществ в конкретном городе.

#### `GET /community/:id/managers`
Назначение:

- вернуть и настройки сообщества, и профили администраторов в одном ответе.

### Utility endpoint’ы

#### `GET /utils/cities`
Назначение:

- прокси к `database.getCities`.

#### `POST /utils/shorten`
Назначение:

- прокси к `utils.getShortLink`.

#### `POST /upload`
Назначение:

- загрузить медиафайлы в локальное хранилище.

Возвращает:

- URL файла;
- тип файла.

## Доменная модель

### `User`
Минимальная локальная модель пользователя VK.

Назначение:

- привязать посты и пины к внутреннему `authorId`;
- не работать везде напрямую со строковым `vkId`.

### `Community`
Центральная модель группового контекста.

Хранит:

- `id` группы;
- токен для публикации;
- название и аватар;
- геопривязку;
- accepted tags;
- расписание публикаций;
- ответственного администратора;
- список админов для уведомлений.

### `Post`
Основная контентная сущность проекта.

Хранит:

- текст;
- тег;
- медиа;
- статус модерации;
- дату публикации;
- `vkPostId`;
- геоконтекст;
- автора;
- связь с `Community`.

### `RadarPin`
Географическая сущность радара.

Может быть:

- ручным пином;
- автоматически созданной точкой из поста.

Хранит:

- тип;
- координаты;
- заголовок;
- описание;
- срок жизни;
- автора;
- связь с постом.

## Основные бизнес-флоу

### Флоу 1. Ручной пин на радаре

1. Пользователь открывает `Radar`.
2. Клиент загружает `/radar/pins`.
3. Пользователь создает событие на карте.
4. Фронтенд вызывает `POST /radar/pins`.
5. Backend создает `User`, если нужно, и затем создает `RadarPin`.

Итог:

- пин появляется на карте как самостоятельная сущность.

### Флоу 2. Создание объявления

1. Пользователь открывает `CreateAd`.
2. Заполняет форму.
3. Фронтенд вызывает `POST /posts`.
4. Backend:
   - гарантирует наличие `User`;
   - определяет целевые сообщества;
   - создает `Post` для каждой целевой группы;
   - при некоторых тегах и координатах создает связанный `RadarPin`;
   - может отправить уведомления администраторам.

Итог:

- в системе появляется материал на модерацию;
- иногда одновременно появляется точка на карте.

### Флоу 3. Модерация

1. Администратор открывает `Moderation`.
2. Фронтенд вызывает `/posts/moderation`.
3. При approve/reject клиент вызывает `PATCH /posts/:id/status`.
4. Backend:
   - меняет статус;
   - может подобрать слот публикации;
   - при наличии токена сообщества публикует пост в VK.

Итог:

- пост получает статус;
- при approve может стать реальной публикацией на стене VK.

### Флоу 4. Публикация в VK

При одобрении поста backend:

1. подготавливает текст публикации;
2. добавляет служебные части:
   - автора;
   - чат;
   - дежурного администратора;
   - геотег;
   - служебный hashtag объявления;
3. загружает локальные медиа в VK;
4. вызывает `wall.post`;
5. сохраняет `vkPostId`.

### Флоу 5. Настройки сообщества

1. Администратор открывает `AppSettings`.
2. Клиент получает данные через `/community/:id/managers`.
3. Администратор меняет город, дежурного, расписание и другие параметры.
4. Фронтенд вызывает `PATCH /community/:id`.
5. Backend обновляет `Community`.

Итог:

- настройки сообщества начинают влиять на радар, модерацию, уведомления и публикацию.

## Основные сильные стороны проекта

- Уже есть рабочая вертикаль `frontend -> backend -> VK API -> БД`.
- Домен проекта понятен и не расплывается.
- Модель `Community / Post / RadarPin / User` хорошо покрывает текущий продуктовый сценарий.
- Фронтенд и backend уже реально интегрированы между собой, а не существуют как отдельные заготовки.
- Поддерживается community context, что хорошо соответствует природе VK Mini Apps.

## Основные архитектурные проблемы

Ниже перечислены проблемы в порядке приоритета.

### 1. Слабая и опасная auth-модель в dev-конфигурации
Главная проблема находится в `backend/src/shared/middleware/vkAuth.ts`.

Сейчас:

- при отсутствии `VK_APP_SECRET` используется `test_secret_key`;
- в этом режиме валидация подписи может быть пропущена;
- `vk_viewer_role` может подставляться как `admin`.

Риск:

- API можно случайно оставить в небезопасном состоянии;
- доверие к ролям становится чрезмерным;
- dev-компромисс начинает влиять на безопасность.

### 2. Вся бизнес-логика слишком сконцентрирована в `routes.ts`
Проблема:

- один файл одновременно содержит HTTP-слой, доменную логику, orchestration и VK integration.

Последствия:

- сложно тестировать;
- сложно рефакторить;
- легко породить регрессию;
- высокий риск конфликтов при развитии проекта.

### 3. Авторизация и role checks непоследовательны
Проблема:

- роль местами проверяется нестрого;
- часть endpoint’ов доверяет `vk_viewer_role` слишком прямо;
- `/posts/moderation` сам помечен как недоведенный по правам.

Следствие:

- правила доступа распределены по обработчикам неравномерно.

### 4. Хардкод окружения во фронтенде и upload URL
Проблема:

- `vk_radar_app/src/utils/api.ts` использует `http://localhost:3000/api/vk_post_app`;
- upload тоже возвращает локальные `localhost`-ссылки.

Следствие:

- код плохо переносится между dev, staging и production;
- окружения становятся хрупкими.

### 5. Модель данных слишком свободна
Проблема:

- `Post.status` хранится как свободная строка;
- `RadarPin.type` тоже строка;
- `media` обрабатывается как не до конца нормализованный формат.

Следствие:

- больше шансов на несогласованность данных;
- бизнес-правила фиксируются в коде, а не в схеме.

### 6. Нет формальной схемной валидации входных данных
Проблема:

- payload’ы валидируются вручную и неравномерно.

Следствие:

- сложнее поддерживать API;
- выше риск тихих ошибок;
- тяжелее безопасно расширять контракт.

### 7. Смешение operational-полей и audit-полей
Проблема:

- `PATCH /posts/swap-time` меняет `createdAt`, хотя это audit-поле.

Следствие:

- искажается семантика даты создания;
- сортировка и история начинают зависеть от workaround’ов.

### 8. Перегруженность отдельных frontend-экранов
Самый явный пример — `Radar.tsx`.

В нем смешаны:

- логика карты;
- геолокация;
- запросы к backend;
- хранение пользовательских настроек;
- модалки;
- админские действия;
- работа с маркерами.

Следствие:

- экран трудно сопровождать;
- повышается стоимость любых изменений.

### 9. Несколько UI-веток выглядят частично недоделанными
Признаки:

- `List` обещает более широкий feed, чем реально загружает;
- cross-posting в `AppSettings` отмечен как незавершенный;
- часть профиля выглядит как mock-данные;
- в коде встречаются временно закомментированные блоки.

Следствие:

- граница между готовым и экспериментальным не всегда прозрачна.

### 10. Нет тестовой и миграционной дисциплины
Проблема:

- не видно полноценного набора тестов;
- не видно устойчивого миграционного контура Prisma;
- важный код интеграций не прикрыт контрактными проверками.

Следствие:

- развитие проекта становится рискованнее;
- регрессии тяжелее ловить заранее.

## Что стоит улучшать в первую очередь

### Приоритет 1. Закрыть auth-риски

- убрать небезопасный bypass по умолчанию;
- отделить dev-bypass явным env-флагом;
- перестать подставлять `admin` по умолчанию;
- нормализовать единый `vkContext` внутри запроса.

### Приоритет 2. Убрать хардкод окружений

- вынести API base URL в env;
- сделать публичный upload base URL конфигурируемым;
- убрать прямую зависимость от `localhost` в доменной логике.

### Приоритет 3. Разрезать `routes.ts`

Минимальная цель:

- выделить `posts`;
- выделить `radar`;
- выделить `community`;
- выделить `vkPublishing`;
- выделить `scheduling`.

### Приоритет 4. Ввести схемную валидацию

- описывать payload’ы через `zod` или аналог;
- отделить shape validation от permission checks;
- сделать входные контракты явными.

### Приоритет 5. Нормализовать модель данных

- ввести enum’ы для статусов и типов;
- стандартизовать `media`;
- завести отдельное поле сортировки вместо переписывания `createdAt`.

### Приоритет 6. Упростить frontend data layer

- ввести тонкий слой кэширования и invalidation;
- декомпозировать `Radar.tsx`;
- уменьшить дублирование fetch-логики по панелям.

## Практический итог
На текущем этапе проект уже обладает реальной продуктовой основой:

- есть клиент;
- есть backend;
- есть модель данных;
- есть интеграция с VK API;
- есть доменная логика вокруг модерации, публикации и карты.

Главные риски сейчас находятся не в отсутствии функциональности, а в инженерной зрелости:

- безопасность dev-контура;
- чрезмерная концентрация логики;
- слабая формализация API;
- отсутствие устойчивой тестовой и миграционной базы.

Если продолжать развитие проекта, то разумная стратегия — не переписывать всё сразу, а пройти поэтапно:

1. закрыть auth и конфигурационные риски;
2. стабилизировать backend-контракты;
3. декомпозировать `routes.ts`;
4. затем улучшать frontend-слой и развивать новые функции.
