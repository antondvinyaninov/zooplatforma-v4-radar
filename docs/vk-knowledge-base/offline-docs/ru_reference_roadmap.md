# Roadmap

Актуальная версия API — [5.199](/ru/reference/versions#5.199).

> API ВКонтакте регулярно меняется. Мы добавляем новые методы, обновляем структуру объектов и запускаем целые сервисы, призванные дать разработчикам больше свободы в реализации своих задач.
> 
> Мы стараемся, чтобы изменения никак не сказывались на работоспособности уже отлаженных продуктов, но иногда требуются очень масштабные вмешательства, о которых лучше узнать заранее — и подготовить свое приложение, чтобы оно продолжило стабильно работать в новых условиях.
> 
> В этом разделе мы рассказываем только о серьёзных изменениях, которые ожидают API ВКонтакте. Подробную историю всех изменений можно найти в разделе [Версии API](/ru/reference/versions).

# Архив

## [](<#2020 | Июнь>)2020 | Июнь

### [](<#Жизненный цикл версий API>)Жизненный цикл версий API

От версии к версии в API меняются названия полей, типы данных, структура объектов. Чтобы корректно обрабатывать в своём приложении ответы от методов API, важно знать, какие данные и в каком виде придут от сервера. Указывая номер версии в своём вызове, вы сообщаете серверу, какой формат данных вы ожидаете получить в ответе.

У версий API появился жизненный цикл: версия устаревает через 2 года после выхода.

#### [](<#Как это работало>)Как это работало

После отказа от мажорных версий  3.0 и 4.0 многие разработчики перешли на версию 5.0. Но вызовы методов с данной версией не поддерживают новые параметры, а формат ответа может сильно отличаться от ответа на запрос с [актуальной версией](/ru/reference/versions).

Запросы к API c версией, жизненный срок которой подошёл к концу, перестают поддерживаться. Сервер вернёт ошибку на запрос к API, в котором указана устаревшая версия.

Необходимо было обновить вызовы методов, указав версию API 5.81 или новее, и убедиться, что передаются актуальные параметры.

> С 19 августа 2021 года срок жизни версий ниже 5.41 закончился.
> 
> С 26 августа 2021 года перестали поддерживаться версии ниже 5.61.
> 
> С 2 сентября 2021 года прекратилась поддержка версий ниже 5.81.

Рекомендуем использовать актуальную версию API — 5.131.

## [](<#2020 | Март>)2020 | Март

### [](<#Перевод приложений сообществ на VK Mini Apps>)Перевод приложений сообществ на VK Mini Apps

Приложения сообществ, запущенные в октябре 2016, значительно расширили функциональность групп и публичных страниц. За три года эта платформа стала неотъемлемой частью бизнес-процессов и решала большинство задач администраторов и подписчиков. В основе приложений сообществ был [JavaScript SDK](/ru/sdk/javascript). Однако чтобы оставаться на острие технологий и унифицировать разработку всей экосистемы ВКонтакте, мы приняли решение отказаться от JavaScript SDK в пользу [VK Bridge](/ru/bridge/getting-started).

Для пользователей каталог остался прежним — изменились только технологии внутри приложений.

#### [](<#Как это работало>)Как это работало

Для работы с платформой приложений сообществ ВКонтакте разработчики использовали JavaScript SDK: скрипт подключался в теге `<head>` страницы с приложением. Пользовательский интерфейс приложения работал через [Client API](/ru/sdk/javascript).

После перехода на VK Mini Apps в каталог для сообществ vk.com/community\_apps попадали только приложения, созданные на платформе VK Mini Apps. Приложения, использующие JavaScript SDK, в каталог больше не добавлялись. Приложения, которые интегрированы с платформой через JavaScript SDK, перестали работать. В каталоге размещались только приложения, использующие [VK Bridge](/ru/bridge/getting-started).

Разработчикам нужно было перевести работу приложений на [VK Bridge](/ru/bridge/getting-started).

К примеру, если приложение позволяет работать с виджетами приложений сообществ — необходимо было подключить библиотеку VK Bridge и заменить вызовы [`showGroupSettingsBox`](/ru/sdk/javascript#3.%20showGroupSettingsBox) и [`showAppWidgetPreviewBox`](/ru/sdk/javascript#20.%20showAppWidgetPreviewBox) на события [`VKWebAppGetCommunityToken`](/ru/bridge/VKWebAppGetCommunityToken) и [`VKWebAppShowCommunityWidgetPreviewBox`](/ru/bridge/VKWebAppShowCommunityWidgetPreviewBox) соответственно.

После того как приложение полностью переведено на события VK Bridge, нужно было изменить его тип на VK Mini Apps при помощи специальной кнопки в настройках: она была доступна главным администраторам в приложениях старого типа.

Это позволило приложению остаться в каталоге и сделать переход незаметным для пользователей. Идентификатор приложения и его ключи при этом не менялись. ![](https://sun9-29.userapi.com/CHXc2GvaN5hODt9_3AwXSKgOYkO_6Y_NvOqWpA/5DdkeMdiEL4.jpg "Блок смены типа в настройках приложения сообществ")Блок смены типа в настройках приложения сообществ

## [](<#2019 | Декабрь>)2019 | Декабрь

### [](<#Отказ от Mobile SDK>)Отказ от Mobile SDK

Direct Games запускались как эксперимент в августе 2017 года и за два года эксперимент превратился в полноценную платформу. Технологической основой интеграции мобильных HTML5-игр с приложениями ВКонтакте был Mobile SDK, простой `js`\-файл, по принципу работы схожий с [JavaScript SDK](/ru/sdk/javascript). Однако чтобы оставаться на острие технологий и унифицировать разработку всей экосистемы ВКонтакте для HTML5-продуктов, мы приняли решение отказаться от Mobile SDK в пользу [VK Bridge](/ru/bridge/getting-started).

#### [](<#Как это работало>)Как это работало

Для интеграции мобильной HTML5-игры с платформой ВКонтакте разработчики использовали Mobile SDK: подключали скрипт в тег `<head>` страницы с игрой. В то же время браузер в мобильных приложениях ВКонтакте уже поддерживал [VK Bridge](/ru/bridge/getting-started): на его стороне работает транспорт, преобразующий вызовы из Mobile SDK в вызовы [VK Bridge](/ru/bridge/getting-started).

Модераторы каталога при публикации игр в каталоге не смотрели на технологический механизм интеграции.

Мы прекратили принимать в каталог игры, которые интегрированы с Direct Games через Mobile SDK. В каталоге теперь могли быть опубликованы только игры с интеграцией через [VK Bridge](/ru/bridge/getting-started).

## [](<#2019 | Сентябрь>)2019 | Сентябрь

### [](<#Отказ от Flash во встраиваемых приложениях>)Отказ от Flash во встраиваемых приложениях

Игровая индустрия активно развивается, а технологии, популярные у разработчиков несколько лет назад, уступают место новым, более гибким в разработке и предоставляющим пользователям разнообразный опыт и более глубокое погружение в игровой процесс.

Flash был важной вехой развития игровой индустрии, однако веб-технологии движутся в сторону открытых стандартов. Компания Adobe прекратила поддержку Flash в 2020 году: [https://theblog.adobe.com/adobe-flash-update/](https://vk.com/away.php?to=https%3A%2F%2Ftheblog.adobe.com%2Fadobe-flash-update%2F). Все популярные браузеры, такие как Google Chrome, Mozilla и другие, начали отключать возможность использования плагина Flash.

В 2018 году ВКонтакте [закрыл](/ru/reference/roadmap) возможность загружать SWF-файлы непосредственно на сайт, а после мы прекратили одобрять в каталог игры на Flash в IFrame.

#### [](<#Как это работало>)Как это работало

Игры, запускающиеся на vk.com, работали через IFrame, который загружается с домена разработчика, и связывались с социальными механизмами ВКонтакте через [JavaScript SDK](/ru/sdk/javascript). Команда модераторов игрового каталога не принимала во внимание, на какой технологии работает игра внутри IFrame, Flash это или, например, HTML5.

После отказа от Flash новые проекты, созданные на Flash, уже не могли пройти модерацию в каталог. В требованиях для модерации приложений появился пункт, сообщающий, что новые приложения, размещаемые на vk.com, должны работать на открытых технологиях и запускаться без установки пользователями дополнительных плагинов.

Мы рекомендовали переходить на открытые технологии: HTML5, WebGL и WebAssembly. Так как эти технологии получали активную поддержку со стороны сообщества разработчиков и создателей игровых движков, таких как Unity, Unreal Engine и PixiJS. Игры, созданные на этих технологиях, не требуют от пользователей дополнительных подтверждений для запуска.

Приложения на Flash, одобренные в каталог до 1 сентября 2019 года, продолжили работать в каталоге, однако предпочтение в продвижении со стороны редакторов было отдано проектам, созданным на нативных веб-технологиях. В начале 2020 года ВКонтакте перестал продвигать Flash-игры в каталоге и стал занижать их позиции в выдаче в каталоге в пользу HTML5-проектов.

## [](<#2019 | Май>)2019 | Май

### [](<#Отказ от мажорных версий API 3.0 и 4.0>)Отказ от мажорных версий API 3.0 и 4.0

От версии к версии в API меняются названия полей, типы данных в них, структура объектов. Чтобы корректно обрабатывать в своем приложении ответы от методов API, важно наперед знать, какие данные и в каком виде придут от сервера. Задавая номер версии в своем вызове, вы сообщаете серверу, какой формат данных вы ожидаете получить в ответе.

В 2014 году было запущено глобальное обновление API, затрагивающее почти все методы, объекты и их поля. Но ещё долгое время не могли отказаться от использования устаревших версий, поскольку для нас важно поддерживать совместимость платформы со старыми приложениями, которыми по-прежнему пользуются многие люди. Но необходимость их поддержки накладывает всё большие ограничения на дальнейшее развитие платформы, в связи с чем мы приняли решение отказаться от версий ниже 5.0.

#### [](<#Как это работало>)Как это работало

После отказа версий по умолчанию для сохранения совместимости в приложениях многие разработчики указали версии ниже 5.0. Но мажорные версии 3.0 и 4.0 уже были устаревшими, в них не были поддержаны многие типы объектов, а формат ответа у большинства методов совсем не был похож на тот, что пользователи видели в примерах документации.

Запросы к API c указанной мажорной версией 3.0 и 4.0 перестали поддерживаться. Сервер начал возвращать ошибку на запрос к API, в котором указана версия ниже 5.0.

Структура объектов и названия полей в новой мажорной версии почти всегда отличается от предыдущей.

## [](<#2019 | Февраль>)2019 | Февраль

### [](<#Ограничение Messages API>)Ограничение Messages API

Одна из первостепенных задач для нашей команды — сохранить в безопасности личную информацию каждого пользователя. Поэтому мы подробно рассказывали о принципах защиты данных и о том, какую информацию могут использовать сервисы ВКонтакте: vk.com/data\_protection.

Наша команда не прекращает работу по усилению защиты персональной информации многомиллионной аудитории ВКонтакте. При этом, конечно, одни из самых чувствительных данных — это личные сообщения. Мы хотим не только дать людям больше уверенности в том, что их информация находится под защитой, но и предотвратить любые намеренные или случайные злоупотребления со стороны тех владельцев сторонних приложений, которые не могут обеспечить приемлемый уровень безопасности и не работают по стандартам нашей команды.

Стремясь предоставить наиболее высокий уровень защиты приватности и хранения личной переписки, мы ограничили сторонним разработчикам доступ к API сообщений пользователей.

Это изменение не затронуло бот-платформу в сообществах, пользователей веб-версии vk.com и официальных мобильных приложений ВКонтакте.

#### [](<#Как это работало>)Как это работало

До этого изменения методы для работы с сообщениями пользователей были доступны любым Standalone-приложениям, использующим авторизацию по схеме [Implicit flow](/ru/api/access-token/implicit-flow-user). Приложения могли использовать публичные методы API ВКонтакте для отображения, отправки и других действий с сообщениями.

Мы сделали методы секции [`messages`](/ru/method/messages) доступными по умолчанию только с ключом доступа сообщества. Для разработки мессенджера достаточно было получить в [Поддержке](https://vk.com/away.php?to=https%3A%2F%2Fvk.com%2Fsupport%3Fact%3Dnew_api) тестовый доступ, подразумевающий работу методов секции Messages с ключами администраторов Standalone-приложения. Для получения полного доступа к пользовательскому Messages API приложению достаточно было пройти модерацию.

#### [](<#Как подготовиться к изменениям>)Как подготовиться к изменениям

Если в вашем приложении используются методы для работы с разделом сообщений пользователя, вам необходимо написать в [Поддержку](https://vk.com/away.php?to=https%3A%2F%2Fvk.com%2Fsupport%3Fact%3Dnew_api), чтобы обсудить условия дальнейшего сотрудничества.

Чат-ботов, использующих ключ доступа пользователя для взаимодействия с сообщениями, следует перенести на платформу [для ботов в сообществах](/ru/api/bots/getting-started).

В случае, если ваше приложение не использует методы секции [`messages`](/ru/method/messages) — убедитесь, что [право доступа](/ru/reference/access-rights) `messages` (+4096) не запрашивается в вашем приложении.

## [](<#2018 | Февраль>)2018 | Февраль

### [](<#Bots LongPoll API>)Bots LongPoll API

Чат-боты в сообществах могли получать события двумя способами: [Callback API](/ru/api/callback/getting-started) или с помощью обычного [LongPoll](/ru/api/user-long-poll/getting-started). Для ботов с большой нагрузкой каждый из этих подходов имел определённые недостатки.

#### [](<#Как это работало>)Как это работало

Мы реализовали отдельный LongPoll для чат-ботов. Его отличие от Callback API в том, что вы сами забираете список событий из ВКонтакте, а не наоборот — вам не нужен скрипт на сервере, куда будет приходить каждое случившееся событие. Отличие от обычного LongPoll в том, что Bots LongPoll поддерживает весь набор событий из Callback API, а не только события из переписки. Кроме того, в сравнении обычным LongPoll увеличен максимальный объём выборки хранимых событий.

Мы рекомендовали создателям всех чат-ботов, в особенности высоконагруженных, перейти к использованию нового инструмента. Им не пришлось существенно переделывать логику работы с самими событиями — формат объектов остается тем же, что и в Callback API.

## [](<#2018 | Февраль>)2018 | Февраль

### [](<#Отказ от версии по умолчанию>)Отказ от версии по умолчанию

От версии к версии в API меняются названия полей, типы данных в них, структура объектов. Чтобы корректно обрабатывать в своем приложении ответы от методов API, важно наперед знать, какие данные и в каком виде придут от сервера. Задавая номер версии в своем вызове, вы сообщаете серверу, какой формат данных вы ожидаете получить в ответе.

Если вы не передавали в своем запросе параметр `v`, содержащий номер версии, по умолчанию он считался равным 3.0. Версия 3.0 устарела, в ней не были поддержаны многие типы объектов, а формат ответа у большинства методов совсем не похож на тот, что разработчики видели в примерах документации.

Мы долгое время не могли отказаться от использования этой версии по умолчанию, поскольку для нас важно поддерживать совместимость платформы со старыми приложениями, которыми по-прежнему пользуются многие люди.

#### [](<#Как это работало>)Как это работало

Параметр `v` с номером версии стал обязательным для всех запросов к API. Сервер возвращал ошибку на запрос, в котором не содержится этот параметр.

Если разработчики не указывали номер версии (`v`) в своих запросах к API, им необходимо было проверить ожидаемый синтаксис объектов в своём коде. Обязательно нужно включать в каждый свой запрос параметр `v` с указанием номера версии. Структура объектов и названия полей в версии 5.0 почти всегда отличаются от используемых в 3.0.

Список изменений в последних версиях доступен здесь: [reference/versions](/ru/reference/versions).

### [](<#Новый сервисный ключ доступа>)Новый сервисный ключ доступа

С апреля 2017 года сервисный ключ доступа обязателен для работы с этими методами (не включая методы секции [`secure`](/ru/method/secure)):

*   •
    
    [`newsfeed.search`](/ru/newsfeed.search);
    
*   •
    
    [`wall.search`](/ru/wall.search).
    

С мая 2018 года сервисный ключ доступа обязателен для работы с этими методами:

*   •
    
    [`users.get`](/ru/users.get);
    
*   •
    
    [`wall.getComments`](/ru/wall.getComments);
    
*   •
    
    [`friends.get`](/ru/friends.get);
    
*   •
    
    [`groups.getMembers`](/ru/groups.getMembers);
    
*   •
    
    [`likes.getList`](/ru/likes.getList).
    

С 14 мая 2018 года сервисный ключ доступа обязателен для работы со всеми методами.

#### [](<#Как это работало>)Как это работало

Некоторые методы API не требуют авторизации. Например, [`users.get`](/ru/users.get) — достаточно указать параметры самого метода и не передавать ключ доступа, сервер вернёт корректный ответ. Такие методы называются открытыми.

В то же время методы секции [`secure`](/ru/method/secure) требуют использования специального ключа доступа, который необходимо получать отдельным запросом к oauth.vk.ru, и передачи в запросе защищенного ключа приложения (`client_secret`).

С апреля 2017 года для использования открытых методов и методов [`secure`](/ru/method/secure) сервисный ключ стал обязательным. Нет необходимости делать отдельный запрос к oauth.vk.ru, чтобы получить его. Срок действия ключа не ограничен, но в любой момент его можно обновить, при этом предыдущий ключ будет аннулирован.

Если в приложении используются открытые методы, необходимо изменить связанную с ними логику таким образом, чтобы в запросах передавался параметр `access_token`, содержащий сервисный ключ доступа. Получить или обновить его можно в разделе редактирования приложения.

Сервисный ключ доступа идентифицирует приложение. Все запросы к API, совершённые с использованием ключа доступа, считаются совершёнными от имени вашего приложения. Сервисный ключ доступа можно использовать только для запросов с серверной стороны приложения, его нельзя передавать и хранить на клиенте. На запросы без ключа доступа сервер возвращает сообщение об ошибке.

## [](<#2016 | Декабрь>)2016 | Декабрь

### [](<#Отключение публичного API для аудио>)Отключение публичного API для аудио

С 16 декабря 2016 года мы отключаем публичный API для работы с аудиозаписями. Существующие методы секции `audio` будут недоступны для вызова, кроме методов для загрузки аудиофайлов.

> Ожидаемая дата изменений: 16 декабря 2016 года.

**Аудиозаписи** — один из популярных сервисов ВКонтакте. Возможность слушать любимую музыку очень важна для наших пользователей. Мы хотим сохранить эту возможность и продолжить развитие аудиораздела.

C 2011 года мы начали переход к легальному использованию контента правообладателей. Сотрудничество с крупнейшими студиями позволяет ВКонтакте размещать эксклюзивные композиции, анонсировать новые альбомы исполнителей и улучшать сервис в целом.

При этом мы вынуждены брать на себя определённые обязательства в том, что касается борьбы с нарушением авторских прав. Это означает, что ВКонтакте не может сохранить публичный API аудиозаписей для приложений сторонних разработчиков в его текущем виде.

Это изменение не затронет пользователей веб-версии vk.com и официальных мобильных приложений ВКонтакте. Более того, в ближайшем будущем мы планируем активно совершенствовать аудиораздел в собственных продуктах.

#### [](<#Как это работало>)Как это работало

Методы для работы с аудиозаписями пользователей и сообществ (`audio`) доступны для любых приложений, использующих публичный API ВКонтакте. В ответ на запрос сервер отправляет информацию о композиции, включая название, имя исполнителя, ссылку на mp3-файл. Приложение использует эти данные для отображения, воспроизведения и других действий с аудиозаписями.

#### [](<#Как подготовиться к изменениям>)Как подготовиться к изменениям

Если в вашем приложении используются методы для работы с аудио, необходимо отключить связанную с ними логику. Сервер API будет возвращать сообщение об ошибке в ответ на такие запросы.

Мы понимаем, что в результате ваше приложение может полностью перестать работать или лишиться значительной части своих возможностей. И в абсолютном большинстве случаев мы не сможем предложить альтернативу.
